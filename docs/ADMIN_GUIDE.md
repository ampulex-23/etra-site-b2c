# Гайд по работе с админкой: от заказа до вручения

## Начальная настройка

### Склады (Склад → Склады)

1. **Склад производства** — тип `Производство`, ответственный: Миша
2. **Склад СДЭК** — тип `Логистика (СДЭК)`, ответственный: Дима

---

## Часть 1. Управление остатками

### 1.1. Пополнение при производстве (делает Миша)

Миша произвёл партию товара. Идёт в **Склад → Движения товаров → Создать**:

- **Тип операции:** `🟢 Произведено`
- **Товары:** добавляет все произведённые позиции (можно несколько в одном движении):
  - Биотархун — 100 шт
  - Биолимонад — 50 шт
- **Склад:** `Склад производства`
- **Кто выполнил:** Миша

**Что происходит автоматически:**

- В таблице **Остатки** для каждого товара на «Складе производства» увеличивается `calculated` и `available` на указанное количество
- Если товар — **набор** (isBundle), система автоматически создаёт отдельное движение для каждого базового товара из набора (Мише делать ничего не нужно)
- Поле `inStock` в карточке товара обновляется автоматически (true если available > 0 хотя бы на одном складе)

### 1.2. Перемещение на склад СДЭК (делает Миша)

Миша передаёт товар Диме. Идёт в **Движения товаров → Создать**:

- **Тип операции:** `🔵 Отправлено в СДЭК`
- **Товары:** Биотархун — 30, Биолимонад — 20
- **Склад:** `Склад производства` (откуда)
- **Целевой склад:** `Склад СДЭК` (куда)
- **Статус операции:** `В транзите`

**Что происходит:**

- `Склад производства`: `calculated -= 30/20`, `available` пересчитывается
- `Склад СДЭК`: `inTransit += 30/20` (товар ещё не принят, но уже учтён как «в пути»)

### 1.3. Приёмка на складе СДЭК (делает Дима)

Дима получил товар и подтверждает. Идёт в **Движения товаров → Создать**:

- **Тип операции:** `✅ Принято в СДЭК`
- **Товары:** Биотархун — 30, Биолимонад — 20
- **Склад:** `Склад СДЭК`

**Что происходит:**

- `Склад СДЭК`: `calculated += 30/20`, `inTransit -= 30/20`, `available` пересчитывается

### 1.4. Списание брака (делает Миша или Дима)

Обнаружили 3 битых бутылки. **Движения товаров → Создать**:

- **Тип операции:** `❌ Списание (утрата/брак)`
- **Товары:** Биотархун — 3
- **Склад:** тот, где обнаружен брак
- **Причина:** «Разбиты при транспортировке»

**Результат:** `calculated -= 3`, `available` пересчитывается.

### 1.5. Выдача сотруднику

- **Тип операции:** `👤 Выдача сотруднику`
- **Сотрудник (получатель):** Имя (текстовое поле)
- Работает аналогично списанию: `calculated -= qty`

### 1.6. Реализация на точке продаж

- **Тип операции:** `🏪 Реализация (точка продаж)`
- **Точка реализации:** «Гранат», «Ярмарка» и т.д.
- `calculated -= qty`

## Часть 2. Инвентаризация

### Проведение (делает ответственный за склад)

**Склад → Инвентаризации → Создать:**

1. Выбрать **Склад** (например, Склад производства)
2. **Статус:** `Черновик` (пока заполняете)
3. **Проводивший:** Миша
4. **Позиции** — добавить каждый товар на складе:
   - Товар: Биотархун, Фактический остаток: 95
   - Товар: Биолимонад, Фактический остаток: 48

**Автоматически при сохранении:**

- Поле `Расчётный остаток` заполняется из текущих StockLevels
- Поле `Расхождение` рассчитывается: `actual - calculated`

5. Сменить **Статус** на `Завершена`

**Что происходит при завершении:**

- Для каждого товара с расхождением ≠ 0 автоматически создаётся движение `📋 Корректировка (инвентаризация)`
- `StockLevels.calculated` сбрасывается на фактическое значение
- `StockLevels.actual` обновляется
- `StockLevels.lastInventoryDate` фиксируется
- `Products.inStock` пересчитывается

## Часть 3. Флоу заказа — базовые товары

### Пример: клиент заказал Биотархун ×2 и Биолимонад ×1 через сайт

#### Шаг 1. Заказ создан (автоматически)

Клиент оформил заказ на сайте. В админке в разделе **Магазин → Заказы** появляется:

- **Номер заказа:** автогенерация
- **Статус:** `Новый`
- **Товары:** Биотархун ×2 (цена подставлена автоматически), Биолимонад ×1
- **Итого:** рассчитано автоматически (subtotal − скидка + доставка)

**Что создаётся автоматически:**

1. **Запись в Доставки** (Магазин → Доставки): метод СДЭК, статус `Ожидает отправки`, получатель из данных клиента, адрес из заказа
2. **Запись в Платежи** (Магазин → Платежи): сумма = total заказа, метод из заказа, статус `Ожидает оплаты`
3. В сайдбаре заказа появляются кликабельные ссылки: **Связанная доставка** и **Связанный платёж**
4. **Резервирование товара:** на ближайшем складе с наличием `StockLevels.reserved += qty`, `available -= qty`
5. Если запаса не хватает — в логах предупреждение (заказ всё равно создаётся)

#### Шаг 2. Оплата (автоматически или вручную)

**Вариант А — онлайн-оплата:**
Платёжный шлюз (ЮKassa) присылает webhook → обновляет запись Payment → `status: paid` → хук автоматически:

- Обновляет `order.payment.status = paid`, `order.payment.paidAt`
- Переводит заказ из `Новый` → `В обработке`

**Вариант Б — ручная фиксация:**
Менеджер идёт в **Платежи**, открывает связанный платёж (или кликает ссылку в сайдбаре заказа) и меняет статус на `Оплачено`. Результат тот же.

#### Шаг 3. Обработка (делает менеджер)

Менеджер видит заказ со статусом `В обработке`. Проверяет комплектацию, готовит к отправке.

#### Шаг 4. Передача в СДЭК (делает менеджер)

Менеджер переходит в **Связанную доставку** (ссылка в сайдбаре заказа) и меняет статус на `Передан в службу`.

**Что происходит автоматически:**

- Хук `deliveryAfterChange` видит: метод = cdek, статус = handed_over
- Создаёт заказ в **API СДЭК** с данными получателя, адресом, весом, составом посылки
- Получает из СДЭК: `cdekOrderUuid` и `trackingNumber` (трек-номер)
- Заполняет их в записи Delivery
- **Синхронизирует трек-номер обратно в заказ** (`order.delivery.trackingNumber`)

#### Шаг 5. Отправка (делает менеджер)

Менеджер меняет **статус заказа** на `Отправлен`.

**Что происходит автоматически:**

1. Создаётся **движение товара** `📦 Отправлено клиентам`:
   - Товары: Биотархун ×2, Биолимонад ×1
   - Склад: `Склад СДЭК` (или первый активный типа `logistics`)
   - Связанный заказ: этот заказ
2. `StockLevels.calculated -= qty` на складе СДЭК
3. Снимается резерв: `reserved -= qty`, `available` пересчитывается
4. `Products.inStock` обновляется

#### Шаг 6. Отслеживание (автоматически через СДЭК webhooks)

СДЭК отправляет webhooks при смене статуса:

- `in_transit` → Delivery.status = `В пути`
- `arrived_at_pickup` → Delivery.status = `Прибыл в ПВЗ`
- `delivered` → Delivery.status = `Доставлен`

#### Шаг 7. Завершение

Менеджер ставит заказу статус `Завершён` (или `Доставлен`).

## Часть 4. Флоу заказа — набор

### Пример: клиент заказал «Набор Дегустация» (содержит Биотархун ×1, Биолимонад ×2)

Весь процесс **идентичен** базовым товарам, кроме одного момента:

Когда при статусе `Отправлен` создаётся движение товара — если в заказе есть товар с `isBundle: true`, хук `stockMovementAfterChange` **автоматически раскладывает набор**:

1. Создаётся основное движение: `items: [{ product: Набор Дегустация, quantity: 1 }]`
2. Хук обнаруживает набор → создаёт **дочернее движение**: `items: [{ product: Биотархун, qty: 1 }, { product: Биолимонад, qty: 2 }]`
3. Дочернее движение заполняет поле `Исходный набор` (readOnly ссылка на набор)
4. Остатки списываются с **базовых товаров**, не с набора

**Менеджерам не нужно делать ничего дополнительного** — декомпозиция набора полностью автоматическая.

## Часть 5. Возвраты

Клиент вернул товар. Менеджер создаёт **Движение товара → Создать**:

- **Тип операции:** `📥 Возврат на склад`
- **Товары:** Биотархун — 1
- **Склад:** куда возвращается (Склад СДЭК или Склад производства)
- **Целевой склад:** если нужно указать откуда вернули
- **Связанный заказ:** выбрать заказ клиента
- **Причина:** «Возврат от клиента — не подошёл вкус»

**Результат:** `calculated += 1`, `available` пересчитывается.

Если нужно вернуть деньги — открыть **Связанный платёж** → изменить статус на `Возврат`.

---

## Часть 6. Отмена заказа

Менеджер меняет статус заказа на `Отменён`.

**Автоматически:**

- `StockLevels.reserved -= qty` для каждого товара
- `available` увеличивается обратно
- Товар «возвращается» в доступные остатки

---

## Формула остатков

| Поле | Описание |
|------|----------|
| **calculated** | Расчётный остаток (сумма всех приходов − уходов) |
| **reserved** | Зарезервировано под заказы (снимается при отправке или отмене) |
| **inTransit** | В пути между складами |
| **available** | `calculated - reserved - inTransit` (то, что реально можно продать) |
| **actual** | Фактический (заполняется только при инвентаризации) |
