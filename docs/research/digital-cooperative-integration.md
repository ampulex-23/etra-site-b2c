# Интеграция с «Цифровым Кооперативом» (ЦК «Восход»): отчёт

_Дата: 2026-04-27_  
_Источники: открытые материалы платформы «Кооперативная Экономика», документация ЦК, репозитории на GitHub `coopenomics/*`._

## 1. Что такое «Цифровой Кооператив» и при чём здесь «Восход»

**«Цифровой Кооператив» (ЦК)** — это федеративная платформа автоматизации потребительских кооперативов, действующих по закону РФ № 3085-1 «О потребительской кооперации». Платформа стандартизирует первичный документооборот пайщика и кооператива (вступление, взносы, заказы, голосования), переводя каждое действие в подписанный электронной подписью PDF + криптографический хэш, фиксируемый в блокчейне.

Архитектурно платформа делится на три слоя:

- **Стандарты** — смарт-контракты на C/C++, описывающие методологию кооперации (правила вступления, типы взносов, голосования и т. д.). Они — «общий знаменатель» для всех подключённых кооперативов.
- **Магазин приложений** — клиентские модули (Web/desktop), реализующие конкретные пользовательские сценарии поверх стандартов (рабочие столы, формы, кнопки).
- **Подсети** — изолированные копии блокчейна и стандартов под отдельные правовые/валютные контуры. В каждой подсети — свой набор союзов и кооперативов.

**ПК «Восход»** — потребительский кооператив, выступающий **оператором** российской подсети ЦК. Подключаемый кооператив юридически вступает в ПК «Восход» (как пайщик-юрлицо), подписывает соглашение о подключении, после чего «Восход» автоматически разворачивает на своих серверах полный комплекс ПО (блокчейн-нода + клиентское приложение + мобильное приложение) на делегированном поддомене за ~24 часа.

Дополнительное обязательное условие членства в подсети РФ — вступление в **Союз потребительских обществ «Русь»** как модератора уставных документов.

## 2. Технологический стек

Подтверждено по репозиториям организации `coopenomics` на GitHub:

| Слой | Реализация | Источник |
|---|---|---|
| Блокчейн-нода (`coopos`) | **Форк `AntelopeIO/leap`** (бывш. EOSIO) на C++20 с расширениями для кооп-экономики | `github.com/coopenomics/coopos` |
| CDT (тулчейн смарт-контрактов) | Форк `AntelopeIO/cdt`, контракты пишутся на C/C++ | `github.com/coopenomics/cdt` |
| Клиент-платформа `mono` | TypeScript / Node.js 20, pnpm 9, Docker, WeasyPrint для PDF | `github.com/coopenomics/mono` |
| Контракты / стандарты | C++ (CDT), хранятся и исполняются в блокчейне | `mono/components/contracts` |
| Backend клиента | `controller` + `parser` (Node.js) | `mono/components/{controller,parser}` |
| Frontend | `desktop` (web-приложение) | `mono/components/desktop` |
| SDK для разработчиков | `@coopenomics/sdk` (TypeScript) | `mono/components/sdk` |
| Уведомления, мессенджер | `notifications`, прокси к `matrix-livekit` (Element/Matrix-стек) | `mono/components/notifications`, `coopos-livekit-proxy` |
| Лицензия | Открытый исходный код (форк EOSIO/Antelope обычно под MIT) | репозитории |

**Ключевой вывод:** COOPOS — это, по сути, приватная Antelope-сеть (DPoS, BFT-финализация, ~0.5s блоки, бесплатные транзакции в рамках выделенного CPU/NET) с предметно-ориентированными смарт-контрактами на C++. Это зрелая, боевая технология (EOSIO/Antelope с 2018 г., в проде у EOS, WAX, Telos, UX Network).

## 3. Как устроен документооборот

Каждое юридически значимое действие — это **смарт-контракт + кнопка в UI**:

1. Пайщик нажимает кнопку → клиент генерирует PDF по шаблону (через WeasyPrint) и подписывает его простой ЭП пайщика (ключ хранится на устройстве).
2. Считается криптографический хэш (SHA-256) PDF.
3. Хэш отправляется в транзакцию смарт-контракта; контракт проверяет условия (роли, кворум, последовательность шагов) и фиксирует хэш в блокчейне.
4. PDF остаётся в локальном реестре кооператива (на сервере оператора, в БД), в блокчейн уходит **только хэш** — это решает вопросы 152-ФЗ (персональные данные не покидают периметр кооператива).
5. Завершённый процесс собирается в «пакет» документов — единая последовательность для аудита/налоговой/суда.

Это гибридная модель: **on-chain — только криптографические доказательства**, **off-chain — все ПДн и сами PDF**. С точки зрения 152-ФЗ это удобно, потому что блокчейн не содержит ПДн и не нуждается в локализации (хотя ноды и так в РФ).

## 4. Два варианта интеграции

### 4.1. Вариант A — работать через оператора ЦК «Восход»

Это путь, на который рассчитана платформа «из коробки».

**Что нужно сделать:**

1. Юридически оформить ETRA как потребительский кооператив (или подключить уже существующий ПК).
2. Вступить в **Союз потребительских обществ «Русь»**, скорректировать устав по их методологии.
3. Подать онлайн-заявку в ПК «Восход», подписать соглашение о подключении ЭП.
4. Делегировать поддомен (например, `coop.etra.ru`) на серверы оператора.
5. Дождаться автоматического деплоя (≈24 ч), пройти инициализацию совета (председатель + члены).
6. 30 дней — период адаптации документооборота, помощь представителя СПО «Русь» через Оп!Кооп.
7. Со 2-го месяца — оплата по факту использования.

**Стоимость (по публичной оферте):**

- Разовый платёж при подключении (сумма на сайте ЛК, требует уточнения).
- Ежемесячный членский взнос: **от 4 500 ₽/мес**, в базу входит 50 новых пайщиков и 50 пакетов документов в месяц. Сверх квоты — отдельно. Доп. приложения из каталога — отдельно.

**Что мы получаем:**

- Готовая блокчейн-нода (управляемая «Восходом»).
- Веб-приложение для пайщиков на нашем поддомене.
- Мобильное приложение пайщика.
- Автоматическое обновление ПО.
- Юридическая защита через СПО «Русь» и юристов «Восхода».
- Доступ к магазину кооперативных приложений и общей сети других ЦК (мессенджер Оп!Кооп, межкооп-расчёты).

**Интеграция с нашим B2C-стеком (Payload + Next.js):**

- Через `@coopenomics/sdk` (TypeScript-пакет) из нашего Next.js — подписывать действия пайщика и отправлять транзакции в блокчейн от его имени.
- Через REST API клиента (`controller`) — двусторонний обмен документами, реестрами пайщиков, статусами заказов.
- Через **smart contract action** на стороне блокчейна — наши собственные контракты могут читать/писать состояние кооператива.
- Webhooks/коллбэки от `parser` → наши endpoint'ы (по аналогии как сделан `coopos-livekit-proxy`).

**Плюсы:**

- Минимум инфраструктурной ответственности.
- Юридическая защита и адаптация документов «под ключ».
- Запуск за дни, а не месяцы.
- Совместимость с экосистемой (другие ЦК, СПО «Русь», магазин приложений) автоматическая.
- ~5 тыс. ₽/мес — пренебрежимо мало для бизнеса.

**Минусы / риски:**

- **Зависимость от оператора:** SLA, работоспособность, инфраструктура — всё на «Восходе». Если он падает, падают все его кооперативы.
- **Контроль над данными:** PDF и ПДн физически на серверах оператора (хотя юридически — наши). Это требует DPA/соглашения.
- **Нет влияния на стандарты** (только через участие в разработке через «Благорост»).
- **Кастомная логика** ETRA в стандартах — только через PR в их репозитории + согласование с союзом. Длинный цикл.
- **Лимит 50/50** в базовом тарифе — для активного B2C мало, придётся платить за overage.

### 4.2. Вариант B — поднять собственную ноду / стать своим оператором

Платформа явно продекларировала эту возможность: «исходный код опубликован на GitHub, любой кооператив имеет техническую возможность запустить альтернативную платформу COOPOS и быть оператором на ней — в этом наша децентрализация».

**Технически нужно поднять (всё опенсорс):**

1. **Узел `coopos`** (форк Antelope/leap):
   - Сборка из исходников: C++20, CMake 3.16+, LLVM 7-11, ~8 ядер CPU и 16 GB RAM на сборку.
   - Эксплуатация: 2 ядра / 4 GB RAM / 40-160 GB SSD.
   - Либо запуск в Docker (готовый образ есть).
   - Конфиг: producer-keys, peers (если присоединяемся к существующей сети — нужны peers «Восхода»; если делаем **свою подсеть** — генезис-блок и набор продюсеров с нуля).

2. **`mono` клиент:**
   - Node.js 20 + pnpm 9.
   - Docker Compose для инфры (Postgres/Mongo, очередь, parser, controller, desktop, notifications).
   - WeasyPrint для рендера PDF.
   - Поднять `controller` (бэкенд) + `desktop` (фронтенд) + `parser` (синк блокчейна в БД).
   - Прогнать `pnpm run setup` (интерактивный конфиг) и `pnpm run reboot`.

3. **Контракты:**
   - Деплой стандартных контрактов из `mono/components/contracts` (boot-скриптом `@coopenomics/boot`).
   - При желании — патчи/собственные контракты на C++ через `cdt`.

4. **Инфраструктура продюсеров:**
   - Antelope требует минимум 21 producer'а для устойчивого DPoS, или режим с одним продюсером для приватной сети (тестового уровня надёжности).
   - Реалистично для своей подсети — 3-5 продюсеров (сами + партнёры).

**Сложность реализации (оценка):**

| Задача | Объём работ | Уровень сложности |
|---|---|---|
| Поднять одиночную ноду coopos в Docker и подключить к существующей сети | 1-2 дня | низкий |
| Поднять полный стек `mono` рядом, синхронизировать с публичной сетью (read-only обозреватель) | 1-2 недели | средний |
| Стать **дополнительным узлом** существующей подсети «Восхода» (полноценный peer, не producer) | 2-4 недели + согласование с «Восходом» | средний |
| Стать **block producer'ом** в подсети «Восхода» | 1-3 месяца + голосование пайщиков | средне-высокий, **юридический, не технический** |
| Запустить **собственную изолированную подсеть** (свой genesis, свои контракты) | 3-6 месяцев | высокий |
| Поддерживать форк (синхронизация с upstream Antelope/Leap, безопасность, обновления) | постоянно, ~0.3-0.5 FTE C++ инженера | высокая операционная нагрузка |
| Юридический контур (свой Союз, методология, договоры с участниками) | 6-12 месяцев | очень высокий, не наша компетенция |

**Плюсы:**

- Полный контроль над инфраструктурой и данными.
- Возможность модифицировать стандарты под себя (особенно если своя подсеть).
- Нет лимитов и операционных платежей.
- Возможность зарабатывать как оператор (привлекать другие кооперативы).

**Минусы / риски:**

- **Высокий технологический порог:** Antelope — нишевая платформа, специалистов на рынке РФ единицы. Поддержка форка C++ кода — серьёзная инженерная работа.
- **Юридический контур своей подсети:** нужен свой союз потребительских обществ, своя методология, своя договорная база — это юр-команда, а не разработка.
- **Сетевой эффект:** в подсети «Восхода» уже есть другие кооперативы, магазин приложений, СПО «Русь». В своей подсети мы будем одни.
- **Окупаемость:** при тарифе 4 500 ₽/мес у оператора, экономика своей ноды (зарплаты SRE/DevOps/C++) сходится только при десятках-сотнях кооперативов на ней.
- **Безопасность:** ответственность за хранение приватных ключей продюсеров, защита от 51% атак (для приватной DPoS — низкий риск, но не нулевой), резервирование.

## 5. Рекомендация

**На горизонте 0-12 месяцев — вариант A (через «Восход»).**

Причины:

1. ETRA — B2C e-commerce, не инфраструктурный провайдер. Мы получаем 80% ценности (юр. документооборот, ЭП, межкооп-связь, мобилка) за 4 500 ₽/мес и пару недель работы.
2. Технология (Antelope) и весь клиентский стек — открытые. Vendor lock-in минимален: при необходимости мы всегда сможем мигрировать на собственную ноду — данные перенесутся, потому что блокчейн один и тот же.
3. Интеграция нашего Payload-стека с ЦК сводится к подключению `@coopenomics/sdk` и нескольких REST/webhook-эндпойнтов — это 2-4 спринта, а не годовой проект.

**На горизонте 12-24 месяцев — гибридный сценарий:**

- Поднять собственную **read-only ноду coopos** (peer без права создания блоков) — это даёт независимый источник истины, верификацию хэшей и снижает зависимость от API оператора. Стоимость ≈ 1 VPS + 0.1 FTE DevOps.
- Если бизнес ETRA вырастет в инфраструктурного игрока в кооперативной экономике — рассматривать переход в block-producer-ы текущей подсети «Восхода» (это политическое решение пайщиков подсети, не техническое).

**Свою отдельную подсеть** имеет смысл рассматривать только если:

- появится стратегический партнёр уровня крупного союза/региона;
- появятся несовместимые с СПО «Русь» требования к стандартам;
- нужна изоляция от других кооперативов (правовая, валютная).

Иначе это переоптимизация без бизнес-ценности.

## 6. Ближайшие технические шаги для интеграции (Вариант A)

1. **Юр-трек** (параллельно): подача заявки в ПК «Восход», вступление в СПО «Русь», корректировка устава.
2. **Тест-трек:** доступ к публичной testnet `https://testnet.coopenomics.world` — изучить SDK, протестировать сценарии (вступление пайщика, оформление взноса).
3. **Прототип интеграции** в `etra-site-b2c`:
   - Установить `@coopenomics/sdk` в Next.js.
   - Реализовать связку `customer` (Payload) ↔ `пайщик` (ЦК): на регистрации/первой покупке предлагать вступить в кооператив, через SDK создавать подписанное заявление.
   - Хук на `Orders.afterChange` → формирование пакета первичных документов в ЦК (заказ-договор-акт) с хэшированием в блокчейне.
   - Админ-панель Payload: вкладка «ЦК» в `Customers` со ссылкой на профиль пайщика и пакетом документов.
4. **Боевой деплой:** после оформления юр. контура — перенос на продовый поддомен, миграция активных клиентов в пайщики.
5. **Мониторинг:** поднять собственную read-only `coopos` ноду для независимой проверки хэшей (можно отложить до момента, когда транзакций станет много).
